% Section 6.3: 不足与未来工作

本文在统一协议下完成了 10 组对照，但受限于算力预算、数据规模与研究时长，仍存在若干可以进一步扩展的方向。本节按「本文的局限」与「后续工作」两层分别列出。

\paragraph{（1）本文研究的局限}
\begin{itemize}
    \item \textbf{基座单一}：仅围绕 CodeGeeX4-ALL-9B（$\approx$9B 参数级）展开，训练 $\times$ 推理的交互结论是否在不同规模（如 1.5B / 14B）与不同家族（如 DeepSeek-Coder、StarCoder）基座上成立，本文未给出直接证据；
    \item \textbf{数据域单一}：语料来自 KodCode 派生样本，偏重算法题；在工程型编码任务（API 调用、系统脚本、框架适配）上的表现留作后续工作；
    \item \textbf{$k$ 值灵敏度未系统化}：本文固定 \texttt{FS\_K=2} 以控制 10 组实验规模，更大 $k$ 下的 few-shot 收益曲线及其与 \texttt{MAX\_LENGTH} 的相互作用尚未展开；
    \item \textbf{评测指标单一}：仅报告 pass@$k$，未涉及代码质量（复杂度、风格一致性）与计算效率（生成延迟、显存占用）维度；
    \item \textbf{多 seed 规模有限}：受算力限制，多 seed 复跑仅对少数锚点组实施，部分差分量的统计显著性尚不能强断言。
\end{itemize}

\paragraph{（2）方法层面的后续工作}
\begin{itemize}
    \item \textbf{混合推理策略}：若第五章得出「$\mathrm{H7} \gtrsim \mathrm{H8}$ 与 $\mathrm{H9} \gtrsim \mathrm{H10}$」同时成立，可考虑在推理时按题目难度动态切换 Direct / CoT-ZS / CoT-FS 三档，用分类器预测「是否值得走两阶段」以节省算力；
    \item \textbf{自适应 few-shot 数}：放开「训练/推理 $k$ 匹配」的硬约束，转而用题目侧信号（强标签密度、题目长度）动态调节 $k$，用混合训练目标覆盖多档示例数；
    \item \textbf{更高效的 CoT 训练目标}：在 assistant 段内对推理与代码两部分加权，或引入 DPO/SimPO 等偏好学习，进一步放大本文 \texttt{LoRA\_cot\_*} 分支的增益。
\end{itemize}

\paragraph{（3）系统层面的后续工作}
\begin{itemize}
    \item \textbf{推理-执行闭环}：把 4.2.2 节子进程执行器接入 beam 层（例如基于单测失败回推的 self-debug），使模型在采样过程中获得执行反馈；
    \item \textbf{更大数据规模}：把 3.2 节的 \texttt{HEAD\_N=10000} 扩到 \texttt{10$^5$} 级，并在 3.3 节的标注链路上替换为更强教师模型，考察 LoRA 对语料规模的增益曲线；
    \item \textbf{工程化部署}：将本文的训练-评测流水线封装为可部署服务（按题目提交、按 pass@$k$ 返回多候选），对接在线判题与 IDE 插件，使研究结论进一步产品化。
\end{itemize}
